# Архитектура чат-бота MAX на вычитку

## 1. Что берём в работу

По ТЗ наша зона ответственности:

- чат-бот в мессенджере MAX;
- сценарии диалога с заявителем;
- интеграция с существующим серверным контуром сервиса;
- доставка уведомлений пользователю о смене статусов сообщения.

Не берём в работу:

- проектирование основной БД сервиса;
- веб-интерфейс администратора и главного администратора;
- экспорт в Excel и административные CRUD-экраны.

## 2. Что важно из ТЗ

Бот должен поддерживать:

- главное меню;
- показ информации "о боте";
- показ юридической информации;
- показ списка нарушений;
- сценарий подачи сообщения о нарушении;
- список "мои сообщения";
- просмотр карточки ранее отправленного сообщения;
- входящие уведомления о смене статуса;
- сценарий ответа на запрос уточнения от администратора.

Ключевые ограничения:

- телефон: 10 цифр;
- адрес: до 1000 символов;
- время/период: до 100 символов;
- описание: до 3900 символов;
- доп. информация: до 3900 символов;
- вложения: фото/видео, до 5 шт.;
- хранение сообщений: не более 30 дней.

## 2.1 Что уточнилось по дополнительным файлам проекта

Из папки `Desktop/maxbot/koap-bot` подтвердились важные вещи:

- есть готовая SQL-схема именно под чат-бот;
- уже зафиксированы справочники категорий и муниципалитетов;
- для MAX выделен отдельный внешний домен API: `violations-collecting-max-api.volganet.ru`;
- административная часть и общий сервис вынесены на другой домен: `violations-collecting.volganet.ru`.

Практический вывод:

- бот лучше держать как отдельный API-контур;
- справочники можно не хардкодить, а получать из backend в том же порядке, в каком они уже заведены в БД;
- модель состояний и статусов уже частично определена схемой данных и её важно не сломать при реализации.

## 3. Что подтверждает API MAX

По документации MAX:

- для production рекомендуется только Webhook, long polling нужен для dev/test;
- webhook должен быть доступен по HTTPS на 443 порту и отвечать `200 OK` не дольше 30 секунд;
- лимит на `platform-api.max.ru`: до 30 rps;
- бот может отправлять сообщения через `POST /messages`;
- бот может отвечать на нажатия inline-кнопок через `POST /answers`;
- доступны кнопки `callback`, `message`, `request_contact`, `request_geo_location`, `link`, `open_app`;
- текст исходящего сообщения ограничен 4000 символами;
- для исходящих файлов есть отдельный `POST /uploads`.

Вывод: архитектуру лучше строить вокруг webhook + отдельного слоя сценариев, а любые тяжёлые операции выносить из HTTP-обработчика событий.

## 4. Предлагаемая схема

```text
MAX Webhook
   |
   v
Transport Layer
HTTP handler -> update parser -> auth/secret check -> dedup
   |
   v
Dialog Application Layer
router -> scenario engine (FSM) -> validators -> response builder
   |                            |
   |                            +-> session store
   |
   +-> MAX API client
   |
   +-> Service API client
   |
   +-> async jobs / outbox
                |
                v
         notifications / retries / heavy operations
```

## 5. Компоненты

### 5.1 Transport Layer

Задача слоя:

- принять webhook от MAX;
- проверить `X-Max-Bot-Api-Secret`, если он используется;
- распарсить `Update`;
- быстро вернуть `200 OK`;
- передать событие в обработку.

Рекомендация:

- не выполнять бизнес-логику прямо в HTTP handler;
- handler должен только валидировать, логировать, дедуплицировать и ставить update в внутреннюю очередь.

### 5.2 Scenario Engine

Сердце решения. Это конечный автомат по состояниям из ТЗ.

Базовые состояния:

- `about`;
- `legal_info`;
- `main_menu`;
- `violations_list`;
- `report_consent`;
- `report_category`;
- `report_municipality`;
- `report_phone`;
- `report_address`;
- `report_period`;
- `report_description`;
- `report_media`;
- `report_extra_info`;
- `report_confirm`;
- `report_accepted`;
- `my_reports_list`;
- `my_report_details`;
- `awaiting_clarification_reply`;
- `fallback_unknown`.

Для каждого состояния описываются:

- допустимые входы;
- текст/клавиатура ответа;
- валидация;
- переходы;
- side effects.

Рекомендация по реализации:

- хранить FSM как явную таблицу переходов, а не как набор вложенных `if/switch`;
- тексты и кнопки вынести в отдельные шаблоны;
- валидации оформить отдельными функциями.

### 5.3 Session Store

Боту нужно где-то хранить промежуточное состояние диалога:

- текущий state;
- draft сообщения;
- список вложений;
- ожидается ли ответ на уточнение;
- id сообщения сервиса;
- служебные флаги идемпотентности.

Предпочтительный вариант:

- `Redis` для сессий и дедупликации webhook-событий.

Если по условиям проекта нельзя вводить отдельное хранилище со стороны бота, второй вариант:

- хранить draft/state во внешнем backend API сервиса.

Но без внешнего хранения bot state будет теряться при рестарте, поэтому in-memory хранение для production не подходит.

Важно: найденная SQL-схема уже предполагает хранение состояния пользователя и черновика на стороне backend:

- `users.max_id` как связка с пользователем MAX;
- `users.stage` как глобальное состояние пользователя;
- `messages.status = 'draft'` для черновика;
- `messages.stage` как шаг заполнения сообщения.

Это означает, что у нас есть два реалистичных варианта:

- хранить session state в Redis на стороне бота;
- использовать backend как source of truth для state/draft.

С учётом найденной схемы я бы считал backend-хранение основным вариантом, а Redis оставил для deduplication, rate limiting и краткоживущих технических ключей.

### 5.4 Service API Client

Так как БД и фронт не входят в нашу часть, бот должен работать через контракт с существующим backend.

Нужны минимум такие методы backend API:

- получить справочник категорий;
- получить справочник муниципалитетов;
- создать сообщение о правонарушении;
- получить список сообщений пользователя за 30 дней;
- получить карточку сообщения по номеру;
- записать ответ пользователя на запрос уточнения;
- принять событие о входящем статусе для отправки пользователю;
- при необходимости принять metadata по вложениям.

Важно: бот не должен знать SQL-схему и не должен писать напрямую в основную БД сервиса.

### 5.5 MAX API Client

Отдельный клиент нужен для:

- отправки сообщений;
- отправки inline-клавиатур;
- ответа на callback;
- при необходимости редактирования ранее отправленных сообщений;
- возможной работы с файлами.

Нужны:

- rate limiting;
- retries с backoff;
- идемпотентность на нашей стороне;
- нормальная типизация payload-ов MAX.

### 5.6 Async Worker / Outbox

Нужен для операций, которые не стоит делать в webhook handler:

- повторные отправки сообщений в MAX;
- отправка уведомлений о статусах;
- отложенная обработка вложений;
- ретраи после временных ошибок MAX или backend;
- очистка просроченных сессий.

## 6. Сценарии взаимодействия

### 6.1 Подача нового сообщения

Поток:

1. Пользователь нажимает `/start` или приходит в меню.
2. Бот показывает меню.
3. Пользователь идёт по сценарию подачи обращения.
4. Scenario Engine валидирует каждый шаг и копит draft.
5. После `отправить` бот вызывает backend API `create report`.
6. Backend возвращает номер сообщения и исходный статус `модерация`.
7. Бот сохраняет связку `max_user_id <-> report_id`.
8. Бот отправляет подтверждение пользователю.

### 6.2 Просмотр "мои сообщения"

Поток:

1. Пользователь открывает список сообщений.
2. Бот запрашивает backend API по `max_user_id`.
3. Backend возвращает список за период хранения.
4. Бот показывает краткий список.
5. Пользователь отправляет номер.
6. Бот получает детальную карточку и показывает её.

### 6.3 Уведомление о смене статуса

Поток:

1. Администратор меняет статус во внешнем сервисе.
2. Backend публикует событие для бота или вызывает bot callback endpoint.
3. Bot notification worker формирует сообщение.
4. Бот отправляет уведомление в MAX.

Поддерживаемые статусы:

- `модерация`;
- `в работе`;
- `запрошена дополнительная информация`;
- `отклонено`;
- `рассмотрено`.

### 6.4 Ответ на запрос уточнения

Поток:

1. Backend сообщает боту, что по сообщению запрошено уточнение.
2. Бот отправляет пользователю текст администратора.
3. Бот переводит пользователя в состояние `awaiting_clarification_reply`.
4. Пользователь отправляет ответ.
5. Бот валидирует длину.
6. Бот передаёт ответ в backend.
7. После подтверждения backend бот возвращает пользователя в меню или в карточку сообщения.

## 7. Модель модулей в Go

Предлагаемое разбиение:

```text
/cmd/bot
/internal/config
/internal/httpserver
/internal/maxapi
/internal/webhook
/internal/scenario
/internal/scenario/states
/internal/scenario/templates
/internal/validation
/internal/session
/internal/serviceapi
/internal/notifications
/internal/queue
/internal/logging
```

### Ответственность пакетов

- `httpserver`: запуск HTTP сервера, health/readiness.
- `webhook`: приём и нормализация Update из MAX.
- `maxapi`: HTTP-клиент MAX API.
- `scenario`: FSM, роутинг событий, переходы состояний.
- `validation`: правила проверки пользовательского ввода.
- `session`: интерфейс хранения state/draft и реализация на Redis.
- `serviceapi`: клиент существующего backend сервиса.
- `notifications`: исходящие сообщения по внешним событиям.
- `queue`: фоновые джобы и ретраи.

## 8. Что можно взять из официального Go-клиента MAX

Репозиторий:

- `github.com/max-messenger/max-bot-api-client-go`

Что в нём полезно для нас:

- готовый клиент к MAX Bot API;
- типы для `Update`, `Message`, attachment-ов и клавиатур;
- парсинг входящих update-ов в типизированные структуры;
- базовый webhook handler;
- методы для `messages`, `subscriptions`, `uploads`.

Как я бы его использовал:

- брать как transport SDK для общения с MAX API;
- обернуть своим адаптером `internal/maxapi`;
- не строить бизнес-логику поверх SDK напрямую во всех слоях приложения.

Почему не стоит завязывать всю архитектуру прямо на библиотеку:

- это thin client, а не полноценный bot framework;
- FSM, session management, интеграция с backend, идемпотентность, outbox и маршрутизацию сценариев он не решает;
- webhook handler из библиотеки слишком низкоуровневый для production-архитектуры и его лучше прятать за нашим `httpserver/webhook` слоем;
- в `go.mod` библиотеки указан `go 1.24.0`, это нужно сразу проверить на совместимость с нашим целевым окружением.

Итоговая рекомендация:

- использовать официальный Go-клиент как основу для MAX transport layer;
- поверх него держать наши интерфейсы, чтобы не привязать доменную логику к внешнему SDK.

## 9. Контракты, которые нужно согласовать с backend

Перед стартом разработки стоит зафиксировать API-контракт между ботом и сервисом.

Минимум нужно согласовать:

- как идентифицируем пользователя: `max_user_id`, `chat_id`, оба поля или внешний `applicant_id`;
- кто хранит draft обращения до отправки: бот или backend;
- как передаются вложения:
  - только metadata/token MAX;
  - либо bot сначала перекладывает файлы в целевое хранилище;
- как backend шлёт событие о смене статуса:
  - HTTP callback в бот;
  - очередь;
  - polling ботом;
- как выглядит номер сообщения для пользователя;
- как backend возвращает тексты статусов, причин отклонения и уточнений;
- как реализуется TTL 30 дней;
- нужна ли повторная отправка пользователю, если MAX временно недоступен.

Что уже видно из SQL-схемы:

- идентификатор пользователя для бота: `users.max_id BIGINT UNIQUE`;
- статусы сообщений в БД: `draft`, `moderation`, `in_progress`, `clarification_requested`, `rejected`, `resolved`;
- глобальные стадии пользователя: `main_menu`, `filling_report`, `viewing_messages`, `waiting_clarification`;
- шаги заполнения сообщения: `category`, `municipality`, `phone`, `address`, `time`, `description`, `files`, `additional`, `sended`.

Из этого следует, что в API-контракте лучше использовать именно эти machine-readable значения, а человекочитаемые подписи формировать отдельно.

## 10. Нефункциональные требования

Нужно заложить сразу:

- идемпотентную обработку webhook-событий;
- structured logging;
- correlation/request id;
- метрики по входящим событиям, ошибкам и времени ответа;
- rate limiting к MAX API;
- graceful shutdown;
- конфиг через env;
- таймауты на все внешние HTTP-вызовы;
- retry policy только для временных ошибок;
- healthcheck и readiness probe.

## 11. Что я бы предложил как MVP

Первый релиз:

- webhook endpoint;
- FSM по всем состояниям из ТЗ;
- Redis для session/dedup;
- backend integration по справочникам, созданию сообщения, списку сообщений, карточке сообщения, ответу на уточнение;
- исходящие уведомления по callback от backend;
- базовые логи, метрики, retry.

Во второй итерации:

- более удобные клавиатуры и callback-навигация;
- шаблонизация текстов из конфигурации;
- админские сервисные команды для диагностики;
- расширенная аналитика;
- резервный polling только для dev.

## 12. Спорные места и мои рекомендации

### Телефон

В MAX есть кнопка `request_contact`, но ТЗ требует ввод "10 цифр без +7/8". Поэтому на вычитку два варианта:

- оставить ручной ввод, строго по ТЗ;
- использовать `request_contact`, а в боте нормализовать номер до 10 цифр.

Практически второй вариант удобнее, но его лучше отдельно согласовать.

### Вложения

ТЗ требует до 5 фото/видео, но не описывает точный backend-контракт по файлам. Это надо утвердить до начала реализации, иначе здесь самый высокий риск переделок.

Дополнение по SQL:

- в таблице `files` пока хранятся `path`, `file_name`, `file_size`, `mime_type`;
- значит backend ожидает уже материализованный файл или ссылку на него, а не просто "файл был приложен".

Здесь нужно заранее решить, кто отвечает за финальное размещение файла:

- бот скачивает/перекладывает файл и передаёт backend путь или object key;
- либо backend сам забирает вложение по данным MAX.

### Юридические тексты и список нарушений

Лучше не зашивать их в код.

Рекомендация:

- получать из backend или хранить во внешнем конфиге/контентном слое.

## 13. Итоговое решение

Я бы предлагал делать бот как отдельный stateless Go-сервис с:

- webhook от MAX;
- явной FSM-моделью сценариев;
- backend как основным хранилищем пользовательских стадий и черновиков;
- Redis для дедупликации, retry-ключей и технического кеша;
- отдельным клиентом MAX API;
- отдельным клиентом backend API сервиса;
- worker-слоем для уведомлений, retry и тяжёлых операций.

Это решение хорошо бьётся с ТЗ, не лезет в БД и фронт, даёт устойчивость к рестартам и не смешивает transport, сценарии и интеграционные контракты в одном коде.
